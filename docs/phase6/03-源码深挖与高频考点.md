```java

@Import(AutoConfigurationImportSelector.class)  // ← 核心：导入配置选择器
public @interface EnableAutoConfiguration {
    // ...
}
```

### 1.3 AutoConfigurationImportSelector

```java
// 核心方法：selectImports()
public class AutoConfigurationImportSelector
        implements DeferredImportSelector, BeanClassLoaderAware, ... {

    @Override
    public String[] selectImports(AnnotationMetadata annotationMetadata) {
        // 1. 获取配置元数据
        AutoConfigurationMetadata autoConfigurationMetadata =
            AutoConfigurationMetadataLoader.loadMetadata(this.beanClassLoader);

        // 2. 加载 spring.factories 中的自动配置类
        // Spring Boot 3.0 后改为 META-INF/spring/xxx.imports
        List<String> configurations = getCandidateConfigurations(annotationMetadata, attributes);

        // 3. 按 @Conditional 条件过滤
        configurations = filter(configurations, autoConfigurationMetadata);

        // 4. 排序
        configurations = sort(configurations, autoConfigurationMetadata);

        return StringUtils.toStringArray(configurations);
    }
}
```

**加载流程**：
```
@EnableAutoConfiguration
  → AutoConfigurationImportSelector
    → 加载 META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports
      → 获取所有 AutoConfiguration 类名
        → 按 @ConditionalOnClass / @ConditionalOnBean / @ConditionalOnProperty 过滤
          → 排序后注册为 Bean
```

### 1.4 条件注解机制

## 二、MyBatis-Plus 核心源码

### 2.1 分页插件原理

```java
// MyBatis-Plus 分页拦截器
public class PaginationInnerInterceptor implements InnerInterceptor {

    @Override
    public void beforeQuery(Executor executor, MappedStatement ms,
                            Object parameter, RowBounds rowBounds,
                            ResultHandler resultHandler, BoundSql boundSql) {
        // 1. 判断是否为分页查询
        IPage<?> page = ParameterUtils.findPage(parameter);  // 参数中是否包含 Page 对象
        if (page == null) return;

        // 2. 自动生成 COUNT 查询（计算总记录数）
        if (page.isSearchCount()) {
            // 改写原 SQL → SELECT COUNT(1) FROM (原SQL) total
            Long total = autoCount(executor, ms, parameter, boundSql);
            page.setTotal(total);
        }

        // 3. 改写原 SQL 加 LIMIT
        // SELECT * FROM table WHERE ... LIMIT offset, pageSize
        String buildSql = dialect.buildPaginationSql(buildSql, page.offset(), page.getSize());
    }
}
```

**面试点**：
> "MyBatis-Plus 的分页不是内存分页，而是拦截 SQL 改写为带有 LIMIT 的物理分页。它会自动生成 COUNT 查询来计算总条数，而且通过方言（Dialect）适配 MySQL、PostgreSQL 等不同数据库的分页语法。"

### 2.2 自动填充（MetaObjectHandler）

```java
// 在 tinybrain-common 中实现
@Bean
public MetaObjectHandler metaObjectHandler() {
```

## 三、Spring Security + JWT 认证流程

### 3.1 认证过滤器链

```
请求到达
  ↓
SecurityContextPersistenceFilter    — 从 Session/Context 中恢复认证信息
  ↓
UsernamePasswordAuthenticationFilter — 处理 /login 表单登录
  ↓
JwtAuthFilter (自定义)               — 从请求头提取 JWT，校验，设置 SecurityContext
  ↓
FilterSecurityInterceptor           — 权限校验（@PreAuthorize、hasRole）
  ↓
Controller 处理请求
```

### 3.2 JwtAuthFilter 实现

```java
public class JwtAuthFilter extends OncePerRequestFilter {

    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                    HttpServletResponse response,
                                    FilterChain filterChain) {

        // 1. 从请求头提取 Token
        String authHeader = request.getHeader("Authorization");
        String token = JwtUtil.extractToken(authHeader);

        // 2. 校验 Token
        if (token != null && JwtUtil.validateToken(token)) {
            Long userId = JwtUtil.getUserId(token);
            String role = JwtUtil.getRole(token);

            // 3. 构建 Authentication 对象，设置到 SecurityContext
            UsernamePasswordAuthenticationToken auth =
                new UsernamePasswordAuthenticationToken(
                    userId, null,
                    List.of(new SimpleGrantedAuthority(role))
                );

            SecurityContextHolder.getContext().setAuthentication(auth);
        }

        // 4. 继续过滤器链
        filterChain.doFilter(request, response);
    }
}
```

**面试点**：
> "因为 JWT 是无状态的，所以 SecurityContextHolder 的信息只在当前请求有效（不需要在 Session 中存储）。每个请求进来，JwtAuthFilter 都会重新解析 Token 构建 Authentication。"

## 四、Spring Cloud Gateway 源码重点

### 4.1 Gateway 处理流程

```
客户端请求
  ↓
DispatcherHandler (WebFlux 核心分发器)
  ↓
RoutePredicateHandlerMapping   — 匹配路由（按 predicates 条件）
  ↓
FilteringWebHandler            — 构建并执行过滤器链

    ↓
    代理请求 → 后端服务
    ↓
    post-filters (响应处理)
  ↓

```

### 4.2 GlobalFilter vs GatewayFilter

```java
// GlobalFilter：所有路由都生效
public interface GlobalFilter {
    Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain);
}

// GatewayFilter：指定路由才生效（yml 中配置）
public interface GatewayFilter extends ShortcutConfigurable {
    Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain);

```

## 六、JVM 必备考点

### 6.1 内存模型

```
线程私有:
  程序计数器     → 当前线程执行的字节码行号
  虚拟机栈       → 栈帧（局部变量表、操作数栈、动态链接、返回地址）
  本地方法栈     → native 方法

线程共享:
  堆 (Heap)      → 对象实例 + 数组（GC 主要区域）
    新生代: Eden / S0 / S1
    老年代: 长期存活的对象
  方法区         → 类信息、常量、静态变量
```

### 6.2 GC 算法对比

| 算法 | 原理 | 适用 | 缺点 |
|------|------|------|------|
| 标记-清除 | 标记存活对象，清除未标记对象 | 老年代 | 产生内存碎片 |
| 标记-复制 | 将存活对象复制到另一半 | 新生代 | 内存利用率低 |
| 标记-整理 | 标记存活对象，向一端移动 | 老年代 | 需要移动大量对象 |
| G1 | Region 分区，优先回收垃圾最多的 Region | 大堆 | GC 停顿可预测 |

### 6.3 类加载机制

```
双亲委派模型：

加载 .class 文件
  → 自底向上检查类是否已加载
    → 自顶向下尝试加载

为什么要双亲委派？ → 安全：防止核心 API 被篡改

ClassLoader 层级：
  Bootstrap ClassLoader（JVM 内置，加载 rt.jar）
    ↓
  Extension ClassLoader（加载 jre/lib/ext/*）
    ↓
  Application ClassLoader（加载 classpath）
```

## 七、MySQL 高频源码理解

### 7.1 InnoDB 存储引擎架构

```
InnoDB 架构：
  Buffer Pool（内存缓冲）
    → 页缓存、Change Buffer、自适应哈希索引
  Redo Log（重做日志）
    → 保证事务持久性（先写日志再写磁盘，WAL）
  Undo Log（回滚日志）
    → MVCC 实现（ReadView + Undo Log 构建快照）

一条 UPDATE 的执行：
  UPDATE user SET name = 'new' WHERE id = 1;

  1. 从磁盘加载 id=1 的数据页到 Buffer Pool
  2. 写入 Undo Log（用于回滚和 MVCC）
  3. 在内存中修改数据页
  4. 写入 Redo Log（prepare 阶段）
  5. 写入 Binlog
  6. Redo Log commit 阶段 → 事务提交
  （Redo Log 两阶段提交确保数据一致性）
```

### 7.2 MVCC 实现原理

```sql
-- 事务隔离级别：RC（读已提交）/ RR（可重复读）
-- MVCC = Multi-Version Concurrency Control

核心组件：
1. Undo Log 版本链：每行记录多个版本
2. ReadView：事务启动时生成"可见性快照"

RR 级别下：
  事务第一次 SELECT 时生成 ReadView
  → 后续所有 SELECT 复用同一个 ReadView
  → 实现"可重复读"

RC 级别下：
  每次 SELECT 生成新的 ReadView
  → 能看到其他事务已提交的修改
```

### 7.3 索引结构：B+Tree

```
B+Tree 特点：
  - 所有数据存在叶子节点
  - 叶子节点形成双向链表（范围查询优势）
  - 非叶子节点只存索引键（扇出大，树高度低）

三层 B+Tree 能存多少数据？
  - 非叶子节点：约 1170 个索引键（16KB / 14B）
  - 叶子节点：每页 16KB
  - 总数据量 ≈ 1170 × 1170 × 16 ≈ 2000 万条
```

### Q1：Spring 的 @Transactional 是怎么通过 AOP 实现的？

```
@Transactional 标注方法
  ↓
Spring 创建 CGLIB 或 JDK 动态代理
  ↓
方法调用 → 代理拦截
  ↓
TransactionInterceptor.invoke()
  ↓
1. 获取事务属性（传播行为、隔离级别等）
2. 检查是否存在当前事务
3. 按传播行为创建/挂起/加入事务
4. try { 执行业务方法 → 提交 }
   catch { 回滚（检查回滚异常类型） }
```

### Q2：Spring Boot 的自动配置和普通 Spring 的 XML 配置有什么区别？

```
普通 Spring：
  <bean id="dataSource" class="com.zaxxer.hikari.HikariDataSource">
      <property name="..." value="..."/>
  </bean>

Spring Boot 自动配置：
  依赖 spring-boot-starter-data-jpa
  → 自动加载 HikariCP + DataSource 配置
  → application.yml 中只需配置 url/username/password

本质区别：Spring Boot 通过条件注解 + 自动配置类，
         在启动时自动判断并创建需要的 Bean。
```

### Q3：ConcurrentHashMap 在 JDK 8 中的改进？

```
JDK 7: Segment + HashEntry + ReentrantLock
  最多 16 个 Segment，锁竞争时可能阻塞

JDK 8: Node + CAS + synchronized
  CAS 插入 TreeBin 的首节点
  冲突时 synchronized 锁单个桶的头节点
  链表 > 8 转红黑树

为什么 JDK 8 性能更好？
  1. synchronized 经 JVM 优化后性能接近 ReentrantLock
  2. 锁粒度从 Segment 级降到桶级
  3. 扩容支持多线程协助（transfer 方法）
```

### Q4：HashMap 的 resize 过程？

```
扩容触发条件：元素数 > 容量 × 0.75
  ↓
1. 新数组 = 旧数组长度 × 2
2. 遍历旧数组每个桶
3. 重新计算元素在新数组的位置：
   元素在新数组中的位置 = 原位置 OR 原位置 + 旧容量
   （利用扩容后 length 的二进制特性，不需要重新 hash）
4. 链表拆分：低位链（原位置）和 高位链（原位置+旧容量）
```
