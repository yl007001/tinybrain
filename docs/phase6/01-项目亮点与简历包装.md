## 三、面试话术模板

### 3.1 "介绍一下你的项目"

```
话术结构：背景 → 技术栈 → 个人贡献 → 难点与亮点

"我主导开发了 TinyBrain，一个个人 AI 知识引擎。
技术选型上使用 Spring Boot 3.2 + Spring Cloud Alibaba 微服务架构，
数据库用 MySQL + Elasticsearch，AI 层对接 DeepSeek/OpenAI 兼容 API。

我主要负责三个核心模块的设计和实现：

第一是 RAG 检索系统。传统问答只能靠 LLM 自身知识，
我们通过'文档分块→向量化→语义检索→增强生成'的链路，
让 LLM 能基于用户私有知识库回答问题。检索准确率的关键在于分块策略和向量模型的选择。

第二是 Agent 智能体系统。我设计了一个基于 Function Calling 的 Agent 引擎，
让 LLM 能自主判断何时调用工具（如搜索知识库、获取时间），
实现了从'被动对话'到'主动行动'的升级。

第三是工程化建设。我搭建了完整的可观测体系：
Prometheus + Grafana 监控业务指标，Zipkin 做链路追踪，
JSON 结构化日志对接 ELK，并通过 Docker Compose 实现一键部署。

这个项目最大的难点是 RAG 系统的检索质量优化，
我通过调整分块大小、重叠窗口和相似度阈值，显著提升了回答准确率。"
```

### 3.2 "项目最大的挑战是什么？"

```
"最大的挑战是 RAG 系统的检索质量问题。
最开始我们发现，用户的提问和文档内容在语义上存在 gap，
直接向量检索往往返回不相关的结果。

我的优化方案是三层递进：
第一层，优化分块策略——500 字符每块 + 100 字符重叠窗口，
在段落边界分割，保证语义完整性。
第二层，加入相似度阈值过滤（>0.3），
低质量匹配不进入后续 LLM 生成阶段，避免误导。
第三层，在 Prompt 层面做了约束——
如果上下文不足以回答问题，必须如实告知，不能捏造。

最终效果是回答的准确率和相关性都有了显著提升。"
```

### 3.3 "项目以后怎么优化？"

```
"如果继续迭代，我会从三个方向优化：

1. 性能方向：当前向量检索是全量扫描 + 线性计算相似度，
   数据量大时会成为瓶颈。可以引入 HNSW 或 IVF 等 ANN 索引，
   将检索复杂度从 O(n) 降到 O(log n)。

2. 检索质量：目前是固定分块策略，
   后续可以引入语义分块（按段落/主题自动分割），
   以及多路召回（向量检索 + BM25 关键词检索 + 重排序），
   综合提升检索效果。

3. 功能方向：Agent 目前只有两个工具，
   可以扩展更多工具（如发邮件、查天气、操作数据库），
   以及引入多 Agent 协作（Supervisor + Worker 模式）。"
```

## 四、面试官可能追问的痛点

| 追问 | 如何应对 |
|------|---------|
| "向量检索为什么用余弦相似度？" | 余弦相似度关注方向而非长度，适合文本语义匹配。欧氏距离受向量长度影响大，不适合。 |
| "JWT 过期了怎么办？" | 短期 Token（7天）+ 刷新令牌机制。或者说结合 Redis 黑名单实现主动失效。 |
| "分块大小 500 怎么来的？" | 经验值 + 实测。块太大超 token 限制，块太小丢失语义。500 字符约 200-300 tokens，是主流实践。 |
| "为什么用 H2 做开发数据库？" | 降低开发环境搭建成本。`MODE=MySQL` 兼容模式确保 SQL 语法一致，上线无缝切换到 MySQL。 |
| "Gateway 和直连有什么不同？" | 网关统一处理鉴权、路由、限流、跨域、日志，各服务不用重复实现。Spring Cloud Gateway 基于 WebFlux，性能优于 Zuul。 |
| "分布式事务怎么做的？" | Phase 1 用本地事务 + 异步解耦。后续可以引入 Seata TCC，但权衡性能与一致性后，当前方案在可接受范围内。 |
| "Micrometer 和 Prometheus 怎么集成的？" | Micrometer 采集指标，通过 `/actuator/prometheus` 暴露，Prometheus 定时拉取。@Timed 注解自动统计方法耗时百分位。 |
| "Docker 多阶段构建的好处？" | 最终镜像只含 JRE + JAR，不包含 Maven 和编译中间产物。体积从 1GB+ 降到 200MB 左右，启动更快，攻击面更小。 |

## 六、面试官眼中的 TinyBrain

### 6.1 为什么这个项目能加分？

| 维度 | 普通项目 | TinyBrain |
|------|---------|-----------|
| 技术栈 | SSM + JSP | Spring Boot 3 + Spring Cloud + Spring AI |
| 架构 | 单体 | 微服务 (Gateway + Nacos) |
| AI 能力 | 无 | RAG + Agent + Function Calling |
| 工程化 | 无 | Prometheus + Grafana + Zipkin + Docker + CI/CD |
| 测试 | 无 | 55+ 单元测试，分层测试策略 |
| 文档 | 无 | 完整的面试向文档，含设计思路和难点分析 |

### 6.2 应届生 vs 有经验者的差异化

> 大部分应届生的项目是"增删改查"，TinyBrain 的差异化在于：
>
> 1. **有 AI** — RAG 和 Agent 是当前最热门的方向
> 2. **有深度** — 不是调 API，而是理解了分块策略、向量检索、Function Calling 的原理
> 3. **有工程** — 微服务、可观测、Docker、CI/CD，体现了工程化思维
> 4. **有体系** — 从底层框架到上层 AI，形成了完整的知识体系
