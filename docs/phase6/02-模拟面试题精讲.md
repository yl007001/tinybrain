## 第一部分：微服务与 Spring Cloud

### Q1：Eureka 停更了，你们为什么选 Nacos？

**答**：Nacos 比 Eureka 的优势有三点：

1. **功能更全面**：Eureka 只有服务注册发现，Nacos 还集成了配置中心（Config），一套系统解决两个问题
2. **支持数据持久化**：Eureka 纯内存，重启后注册信息丢失；Nacos 支持 MySQL 持久化
3. **健康检查更灵活**：Eureka 只支持 Client 心跳，Nacos 支持心跳 + 主动探测

在 TinyBrain 中，我们使用 Nacos Discovery 做服务发现，用 Nacos Config 管理各模块配置（后续可以做到配置热更新）。

### Q2：GateWay 的过滤器和 Servlet Filter 有什么区别？

**答**：

| 维度 | Spring Cloud Gateway | Servlet Filter |
|------|---------------------|---------------|
| 底层 | WebFlux（Netty，响应式） | Servlet API（Tomcat，阻塞 IO） |
| 线程模型 | 事件驱动，少量线程处理大量请求 | 每个请求一个线程 |
| 适用 | API 网关 | Web 应用内部 |
| 扩展 | GatewayFilter + GlobalFilter | Filter 接口 |

**TinyBrain 实践**：我们用 GlobalFilter 做 JWT 全局鉴权，白名单路径放行（login/register），其余请求校验 Token 后将用户 ID 注入请求头传递给下游服务。

**面试加分点**：
> "Gateway 的 GlobalFilter 是有序的，通过 getOrder() 控制执行顺序。我们的 JwtAuthGlobalFilter 设置 order=-100，优先于其他过滤器执行。"

### Q3：服务间调用怎么做的？OpenFeign 的原理？

**答**：使用 OpenFeign 声明式 HTTP 客户端。

```java
@FeignClient("tinybrain-user")
public interface UserServiceClient {
    @GetMapping("/api/auth/me")
    R<UserVO> getCurrentUser(@RequestHeader("X-User-Id") Long userId);
}
```

**原理**：
1. `@EnableFeignClients` 扫描 `@FeignClient` 接口
2. 为每个接口创建 JDK 动态代理
3. 方法调用时，通过 RequestTemplate 构建 HTTP 请求
4. 结合 Ribbon/RibbonLoadBalancerClient 做负载均衡
5. 发送 HTTP 请求，解析响应

### Q4：服务雪崩是什么？怎么防止？

**答**：一个服务故障 → 调用方线程阻塞 → 资源耗尽 → 级联故障。

**TinyBrain 的防护策略**：
1. **超时控制**：Feign 调用设置 connectTimeout 和 readTimeout
2. **限流**：Gateway 层 RequestRateLimiter（基于 Redis），每秒最多 100 请求
3. **熔断**：可以引入 Sentinel，熔断降级 + 限流 + 系统自适应保护
4. **隔离**：线程池隔离，不同服务用不同线程池

**面试加分点**：
> "在微服务架构中，我们认为预防比处理更重要。TinyBrain 在 Gateway 层做了请求限流，确保突发流量不会压垮下游服务。"

## 第二部分：RAG（Retrieval Augmented Generation）

### Q5：RAG 解决了什么问题？和微调（Fine-tuning）有什么区别？

**答**：

| 维度 | RAG | Fine-tuning |
|------|-----|------------|
| 原理 | 检索外部知识 → 拼入 Prompt → LLM 生成 | 用领域数据继续训练模型 |
| 知识更新 | ✅ 即时更新（换文档即可） | ❌ 需要重新训练 |
| 成本 | 低（调 API + 维护向量库） | 高（GPU 训练 + 数据标注） |
| 幻觉控制 | ✅ 强（生成基于检索结果） | ❌ 只靠模型记忆 |
| 适用 | 知识库问答、FAQ、客服 | 风格模仿、特定格式输出 |

**TinyBrain 选 RAG 的原因**：
> "个人知识库的特点是内容频繁更新（用户随时添加/修改文档），RAG 能做到'文档即服务'——新建文档后自动分块向量化，用户立刻就能检索到。如果用微调，每次更新都要重新训练，成本和时间都不可接受。"

### Q6：检索质量怎么衡量和优化？

**答**：检索质量直接影响 RAG 效果，我们从三个维度优化：

**评估指标**：
- Recall@K：前 K 个结果中包含相关文档的比例
- Precision@K：前 K 个结果中相关文档的比例
- MRR (Mean Reciprocal Rank)：第一个相关结果位置的倒数均值

**优化策略**：

```
1. 分块策略（当前）
   固定 500 字符 + 100 字符重叠 → 段落边界分割

2. 多路召回（规划中）
   向量检索（语义） + BM25（关键词） → 融合排序

3. 重排序（规划中）
   用 Cross-Encoder 模型对 Top-K 结果重新打分
   纠正向量检索的语义偏差

4. 查询改写（规划中）
   用户问题 → LLM 改写为更规范的查询 → 检索
   如"Spring事务怎么回事" → "Spring事务的原理和实现"
```

**面试加分点**：
> "我们的相似度阈值设为 0.3，这个值是通过实验确定的。大于 0.3 的结果在语义上基本匹配，低于这个值的匹配会引入噪声，宁愿不返回也不能误导LLM。"

### Q7：为什么选余弦相似度？和其他距离度量比呢？

**答**：

| 度量 | 关注点 | 适合场景 | 值域 |
|------|--------|---------|------|
| 余弦相似度 | 方向夹角 | 文本语义匹配 | [-1, 1] |
| 欧氏距离 | 绝对距离 | 图像、数值特征 | [0, ∞) |
| 点积 | 方向 + 长度 | 经过归一化后等价于余弦 | (-∞, ∞) |

**为什么选余弦**：
> "文本 Embedding 向量的长度（模）通常反映了文本的详细程度，而非语义相似度。余弦相似度只看方向夹角，不管向量长度，正好符合语义匹配的需求。比如一篇详细的文档和一段简短的问题，语义可能是相同的，但欧氏距离会很大，余弦相似度则不受影响。"

## 第三部分：Agent 与 Function Calling

### Q8：Function Calling 的原理是什么？

**答**：本质是 **LLM 输出结构化 JSON → 系统解析执行 → 结果回填**。

**传统 API 调用**：
```
前端硬编码调用 API → 逻辑写死 → 用户无法自定义
```

**Function Calling**：
```
LLM 自主判断 → 生成 {"tool": "knowledge_search", "args": {"query": "..."}} 
  → 系统执行工具 → 结果回填 → LLM 整合生成最终回答
```

**TinyBrain 实现**：

```java
while (iterations < maxIterations) {
    // 1. 将工具描述放入 System Prompt
    String systemPrompt = agentEngine.buildSystemPrompt();

    // 2. LLM 判断是否需要调用工具
    String reply = llmClient.chat(systemPrompt, messages);

    // 3. 解析回复，检查是否包含工具调用
    String toolCallJson = extractToolCall(reply);
    if (toolCallJson == null) break;  // LLM 直接回答了

    // 4. 执行工具，结果回填
    String result = agentEngine.executeTool(toolName, argsJson);
    messages.add(toolResult(toolName, result));
    // 继续下一次循环
}
```

### Q9：怎么防止 Agent 无限循环？

**答**：三个机制：

1. **最大迭代次数**：`maxIterations = 5`，到达上限强制停止
2. **重复调用检测**：连续相同工具调用 → 主动打断
3. **超时控制**：整个 Agent 会话设置超时时间

**面试加分点**：
> "maxIterations 放在 AgentRequest 的 Config 对象中，每个请求可以自定义。管理员调用可以设大一点（复杂任务需要多步工具），普通用户用默认值 5，兼顾灵活性和安全性。"

### Q10：工具注册机制怎么设计的？用了什么设计模式？

**答**：**策略模式（Strategy Pattern）+ SPI 机制**。

```java
// 1. 定义工具接口（策略抽象）
public interface AgentTool {
    String getName();           // 工具名称
    String getDescription();    // 工具描述（LLM 理解用）
    ObjectNode getParametersSchema();  // 参数 Schema
    String execute(JsonNode args, ObjectMapper mapper);  // 执行
}

// 2. 具体工具实现（具体策略）
@Component  // Spring 自动扫描注册
public class KnowledgeSearchTool implements AgentTool { ... }

@Component
public class DateTimeTool implements AgentTool { ... }

// 3. 引擎注册（策略管理）
@Component
public class AgentEngine {
    Map<String, AgentTool> toolRegistry = new ConcurrentHashMap<>();

    public void registerTool(AgentTool tool) {
        toolRegistry.put(tool.getName(), tool);
    }
}

// 4. 启动时自动扫描注册
@Component
public class AgentToolRegistry implements ApplicationListener<ApplicationReadyEvent> {
    @Autowired
    private List<AgentTool> tools;  // Spring 自动收集所有 AgentTool Bean

    @Override
    public void onApplicationEvent(ApplicationReadyEvent event) {
        agentEngine.registerTools(tools);
    }
}
```

**扩展性**：
> "想加一个新工具，只需要新建一个类实现 AgentTool 接口，加上 @Component，Spring 启动时自动注册。不需要修改任何现有代码——开闭原则的最佳实践。"

## 第四部分：可观测性

### Q11：链路追踪的 traceId 是怎么跨服务传递的？

**答**：通过 HTTP 请求头传递。

```yaml
# Micrometer Tracing 配置
management:
  tracing:
    propagation:
      type: w3c  # W3C TraceContext 标准
```

Gateway 收到请求后，Micrometer Tracing 自动在请求头中注入 `traceparent`：
```
traceparent: 00-0af7651916cd43dd8448eb211c80319c-b7ad6b7169203331-01
```

下游服务解析该请求头，提取 traceId，继续传递。

**面试加分点**：
> "traceId 会自动注入到 MDC 中，所以我们在所有日志中都能看到 traceId。配合 Logstash 的 JSON 日志格式，可以在 ELK 中直接按 traceId 搜索，几秒钟就找到了相关联的所有日志。"

### Q12：Micrometer @Timed 注解的工作原理？

**答**：基于 Spring AOP。

```
方法调用 → AOP 切面拦截
  → 记录开始时间（System.nanoTime()）
    → 执行目标方法
      → 记录结束时间
        → 计算耗时
          → 注册到 MeterRegistry（满 buff 的指标体系）
            → Prometheus /actuator/prometheus 暴露
```

@Timed 自动生成：
```
rag_ask_time_seconds_count{...}  # 调用次数
rag_ask_time_seconds_sum{...}    # 总耗时
rag_ask_time_seconds_max{...}    # 最大值
rag_ask_time_seconds_percentile_0.95{...}  # P95
```

**面试加分点**：
> "percentiles = {0.5, 0.95, 0.99} 分别对应中位数、P95、P99。通过 Grafana 看 P99 的波动曲线，能及时发现系统性能拐点。"

## 第五部分：开放性问题

### Q13："如果你要设计一个百万 QPS 的问答系统，怎么设计？"

```
客户端 → CDN → 负载均衡器 → Gateway 集群
  → 知识库分片（按用户 ID 哈希）
    → 缓存层（Redis 缓存热点文档）
      → 向量库分片（按文档 ID 范围分片）
        → LLM 服务（异步 + 池化）

关键点：
1. 多级缓存：Guava Cache（本地）→ Redis（分布式）→ MySQL（持久化）
2. 读写分离：ES 处理搜索，MySQL 做持久化
3. 异步化：文档上传 → MQ → 异步分块向量化
4. 容量规划：单机 200 QPS，5 台机器支撑 1000 QPS，水平扩展
5. 限流降级：超出容量返回缓存结果或友好提示
```

### Q14："Redis 在你的项目里能用在哪些地方？"

| 场景 | 方案 | 说明 |
|------|------|------|
| Token 黑名单 | SET key userId | JWT 主动失效 |
| 接口限流 | Redis + Lua 脚本 | Gateway 的 RequestRateLimiter |
| 缓存热点文档 | String + JSON | 减少数据库查询 |
| 会话记忆 | Hash | Agent 多轮对话（当前是内存实现） |
| 分布式锁 | SETNX + Lua | 重复文档索引防重 |

### Q15："读过 Spring Boot 哪些源码？"

**建议读过的源码（面试高频）：**

1. **@SpringBootApplication 组合注解** → `@EnableAutoConfiguration` → `AutoConfigurationImportSelector` → 加载 `spring.factories`
2. **Spring Boot 启动流程**：`SpringApplication.run()` → `prepareContext` → `refreshContext` → `afterRefresh`
3. **自动配置条件注解**：`@ConditionalOnClass`、`@ConditionalOnMissingBean`、`@ConditionalOnProperty`
4. **@Transactional 原理**：AOP 代理 → `TransactionInterceptor` → 事务管理器 → 提交/回滚
5. **Spring Security 过滤器链**：`FilterChainProxy` → `SecurityFilterChain` → 各个 Filter

**面试话术**：
> "我读过 @SpringBootApplication 的源码，它是个组合注解。核心是 @EnableAutoConfiguration，通过 AutoConfigurationImportSelector 加载 META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports 中定义的所有自动配置类，再按 @Conditional 条件过滤。比如 MyBatis-Plus 的自动配置就依赖了 mybatis-plus-spring-boot3-starter，只有 classpath 中存在相关类时才生效。"

## 第六部分：高频场景题

### Q16："线上 OOM 了，怎么排查？"

```
1. 看监控：Grafana JVM 大盘 → 堆内存曲线
2. 看日志：近期是否有大量请求/超时
3. 生成 Heap Dump：jmap -dump:format=b,file=heap.hprof <pid>
4. 分析：MAT / JProfiler 分析大对象/GC Root
5. 常见原因：
   - 大对象（一次性加载全部数据到内存）
   - 内存泄漏（ThreadLocal 未释放，连接未关闭）
   - 流量突增（扩容前被压垮）
```

### Q17："慢 SQL 怎么优化？"

```
1. 定位：慢查询日志（slow_query_log）→ 获取慢 SQL
2. 分析：EXPLAIN → type、rows、Extra
3. 优化手段（按优先级）：
   a. 加索引（覆盖索引 > 复合索引 > 单列索引）
   b. 优化 SQL（避免函数操作、避免隐式类型转换）
   c. 分页优化（子查询延迟关联）
   d. 读写分离（主库写，从库读）
```

### Q18："接口响应慢，怎么排查？"

```
API Gateway 收到请求
  ↓ 1. Gateway: 响应时间是否正常？
  ↓ 2. Zipkin: 哪个服务慢？哪个方法慢？
  ↓ 3. Prometheus: P99 延迟曲线是否有异常？
  ↓ 4. 日志: traceId 关联查看执行细节
  ↓ 5. MySQL: 慢查询？连接池满？
  ↓ 6. LLM API: 超时？返回慢？
```
