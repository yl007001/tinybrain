## 第一部分：Java 基础

### 1. HashMap 底层原理

**问题**：HashMap put 方法执行流程？

```
put(key, value)
  → hash(key) 计算哈希值（扰动函数）
    → 定位桶位置 (n-1) & hash
      → 桶为空 → 直接插入
      → 桶不为空 → 遍历链表/红黑树
          → 找到相同 key → 覆盖 value
          → 未找到 → 尾插法插入
    → 检查是否达到扩容阈值 → 扩容（resize）
```

**在 TinyBrain 中的应用**：
```java
// Agent 工具注册表：用 ConcurrentHashMap 存储已注册的工具
private final Map<String, Tool> toolRegistry = new ConcurrentHashMap<>();
```

**进阶问题**：
- 为什么容量是 2 的幂？→ `(n-1) & hash` 等价于取模，位运算更快
- 链表转红黑树的阈值？→ 链表长度 >= 8 且数组长度 >= 64
- 什么时候扩容？→ 元素数量 > 容量 × 负载因子（默认 0.75）

### 2. ConcurrentHashMap 线程安全机制

**问题**：ConcurrentHashMap 如何保证线程安全？

| 版本 | 实现方式 |
|------|---------|
| JDK 7 | Segment 分段锁（锁粒度：一个 Segment 多个桶） |
| JDK 8 | **synchronized + CAS**（锁粒度：单个桶） |

JDK 8 改进：
- 使用 synchronized 替代 ReentrantLock（JVM 级别优化）
- 插入时 CAS 检查，失败再锁
- 扩容时多个线程协作（transfer 方法）

### 3. 线程池

**问题**：线程池的核心参数和执行流程？

```java
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    2,          // corePoolSize：核心线程数
    5,          // maximumPoolSize：最大线程数
    60L,        // keepAliveTime：空闲线程存活时间
    TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(10),  // 工作队列
    new ThreadPoolExecutor.CallerRunsPolicy()  // 拒绝策略
);
```

执行流程：
```
提交任务
  → 核心线程 < corePoolSize → 创建新线程执行
  → 核心线程已满 → 放入工作队列
  → 队列已满 → 创建临时线程（直到 maximumPoolSize）
  → 线程数已达上限 → 执行拒绝策略
```

**拒绝策略选择**：
| 策略 | 说明 | 适用场景 |
|------|------|---------|
| AbortPolicy | 抛异常（默认） | 重要任务，不能丢 |
| CallerRunsPolicy | 调用线程自己执行 | 压降（TinyBrain 选择） |
| DiscardPolicy | 直接丢弃 | 不重要的任务 |
| DiscardOldestPolicy | 丢弃最旧的任务 | 消息类场景 |

**在 TinyBrain 中的应用**：
```java
// RAG 并发检索：并行查询多个分块
@Bean("ragExecutor")
public ThreadPoolExecutor ragExecutor() {
    return new ThreadPoolExecutor(
        2, 4, 60, TimeUnit.SECONDS,
        new LinkedBlockingQueue<>(100),
        new ThreadPoolExecutor.CallerRunsPolicy()
    );
}
```

## 第二部分：Spring 全家桶

### 4. Spring IoC 容器

**问题**：Spring IoC 容器初始化流程？

```
ApplicationContext 启动
  → BeanDefinition 加载（XML/注解/配置类）
    → BeanFactoryPostProcessor 处理
      → Bean 实例化（反射创建对象）
        → 属性填充（依赖注入）
          → Aware 接口回调
            → BeanPostProcessor#postProcessBeforeInitialization
              → @PostConstruct / InitializingBean
                → BeanPostProcessor#postProcessAfterInitialization
                  → Bean 就绪
```

### 5. Bean 作用域

| 作用域 | 说明 | TinyBrain 使用 |
|--------|------|--------------|
| singleton | 单例（默认） | Service、Mapper、Controller |
| prototype | 每次创建新实例 | 某些 DTO、事件对象 |
| request | 每个 HTTP 请求一个 | 暂未使用 |
| session | 每个 Session 一个 |（无状态 JWT 不需此作用域） |

### 6. @Transactional 失效场景

**面试必问**：以下场景事务会失效：

| 场景 | 原因 | 解决方案 |
|------|------|---------|
| **私有方法** | CGLIB 动态代理只能增强 public 方法 | 改为 public |
| **同类调用** | this.method() 绕过代理 | 注入自己或拆类 |
| **捕获异常** | 框架捕捉不到异常 | 不要吞异常 |
| **非 RuntimeException** | 默认不回滚 | 设置 rollbackFor |
| **方法不是 public** | AOP 增强限制 | 改为 public |

## 第三部分：MySQL

### 8. 索引相关

**问题**：如何判断 SQL 是否走索引？

使用 `EXPLAIN` 分析执行计划：

| 字段 | 值的含义 |
|------|---------|
| **type** | `system` > `const` > `eq_ref` > `ref` > `range` > `index` > `ALL` |
| **rows** | 扫描行数（越小越好） |
| **Extra** | `Using index`（覆盖索引）、`Using filesort`（需要优化）、`Using temporary`（需要优化） |

**TinyBrain 面试话术**：
> "在我们的文档列表查询接口中，用户按创建时间倒序查看文档。我们在 `user_id` 和 `create_time` 上建立了复合索引，`EXPLAIN` 显示 type=ref，Extra=Using index，实现了索引覆盖，避免了回表查询。"

### 9. SQL 优化案例

```sql
-- 慢查询：扫描了 10 万行
SELECT * FROM kb_document ORDER BY create_time DESC LIMIT 10000, 10;

-- 优化后：使用子查询 + 索引覆盖
SELECT * FROM kb_document
WHERE id > (SELECT id FROM kb_document ORDER BY create_time DESC LIMIT 10000, 1)
ORDER BY create_time DESC LIMIT 10;
```

## 第四部分：项目深度题

### 10. "你为什么做 TinyBrain 这个项目？"

> "市面上虽然有很多知识管理工具（Notion、飞书），但它们都不支持私有化部署，且缺失基于 AI 的 RAG 检索能力。TinyBrain 的定位是 **可自部署的个人 AI 知识引擎**：
>
> 1. **技术深度**：实现了从文档分块、向量化、语义检索到 Agent 编排的完整链路
> 2. **工程实践**：Spring Boot 3 + Spring Cloud Alibaba 微服务架构
> 3. **实用价值**：解决"信息过载 → 智能检索 → 知识生成"的真实痛点"

### 11. "项目最大的技术挑战是什么？"

> **挑战 1：JWT 无状态认证的设计**
> - 问题：无状态的 JWT 无法在服务端主动失效
> - 解决：采用 Token 黑名单（Redis） + 短期 Token（7天） + 刷新令牌机制
>
> **挑战 2：事务一致性与性能的平衡**
> - 问题：创建文档需要同时写 MySQL 和向量库，涉及分布式事务
> - 解决：Phase 1 用本地事务 + 异步队列解耦，后续引入 Seata TCC

### 12. "如果用户量增长 100 倍，项目怎么做架构升级？"

> 1. **数据库**：主从复制 → 分库分表（ShardingSphere）
> 2. **缓存**：Caffeine 本地缓存 → Redis 分布式缓存
> 3. **读写分离**：Elasticsearch 处理搜索，MySQL 做持久化
> 4. **微服务化**：将知识库、RAG、Agent 拆分为独立服务
> 5. **异步化**：文档处理走 MQ（RocketMQ）削峰
> 6. **部署**：K8s 弹性伸缩，按需扩缩容

```

    A原子C一致I隔离D持久
```
