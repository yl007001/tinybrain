# Phase 2：RAG 检索系统实现

## 一、RAG 是什么？

### 1.1 解决的核心问题

LLM（大语言模型）有两大缺陷：
1. **知识截止**：训练数据截止到某个时间点之后的知识不知道
2. **幻觉**：不知道的问题会"编造"答案

**RAG（Retrieval Augmented Generation）** 的解决思路：
```
用户提问
    → 从知识库中检索相关文档片段
        → 将上下文 + 问题拼接成 Prompt
            → LLM 基于上下文回答
                → 有据可查、减少幻觉
```

### 1.2 RAG vs 微调

| 维度 | RAG | 微调 (Fine-tuning) |
|------|-----|-------------------|
| 知识更新 | ✅ 实时更新（换文档即可） | ❌ 需要重新训练 |
| 实现成本 | ✅ 低（无需 GPU 训练） | ❌ 高（需要训练资源） |
| 幻觉控制 | ✅ 强（有上下文约束） | ⚠️ 中等 |
| 对私有数据 | ✅ 天然支持 | ✅ 支持 |
| 推理速度 | ⚠️ 多一步检索 | ✅ 直接生成 |

**TinyBrain 选择 RAG 的原因**：适合个人知识库场景——文档经常增删改，RAG 无需重新训练。

## 二、TinyBrain RAG 架构

### 2.1 系统流程

```
                    ┌─────────────┐
                    │   用户上传文档  │
                    └──────┬──────┘
                           ▼
                    ┌─────────────┐
                    │  DocChunkStrategy │  ← 分块策略
                    │   (500字符+100重叠)│
                    └──────┬──────┘
                           ▼
                    ┌─────────────┐
                    │  LLM Embedding │  ← 调用 API 向量化
                    │  DeepSeek API │
                    └──────┬──────┘
                           ▼
                    ┌─────────────┐
                    │  VectorStore │  ← 内存向量存储
                    │ (余弦相似度)  │
                    └─────────────┘

用户提问:
    ┌──────────────┐
    │  "Spring事务  │
    │   怎么实现的?" │
    └──────┬───────┘
           ▼
    ┌──────────────┐
    │  Embedding   │
    │   (向量化问题) │
    └──────┬───────┘
           ▼
    ┌──────────────┐
    │  VectorStore │  ← Top-K 余弦相似度检索
    │  search()    │
    └──────┬───────┘
           ▼
    ┌──────────────┐
    │  拼接 Prompt  │  ← 上下文 + 问题
    └──────┬───────┘
           ▼
    ┌──────────────┐
    │  LLM Chat    │  ← 调用 DeepSeek Chat
    │  生成回答     │
    └──────┬───────┘
           ▼
    ┌──────────────┐
    │  RAGResult   │  ← 返回答案+来源
    └──────────────┘
```

### 2.2 核心组件

| 组件 | 类名 | 职责 |
|------|------|------|
| 分块策略 | `DocChunkStrategy` | 将长文档切割为语义完整的块 |
| LLM 客户端 | `LLMApiClient` | 封装 Chat + Embedding API 调用 |
| 向量存储 | `VectorStore` | 内存向量库，余弦相似度检索 |
| RAG 服务 | `RAGService` | 编排索引 + 问答全流程 |
| RAG 控制器 | `RAGController` | REST API 接口 |

## 三、分块策略详解

### 3.1 为什么需要分块？

```
[整篇文档] → 直接向量化 → 一个向量
    ❌ 问题："什么是事务？" 与整篇文档相似度低，检索不到

[文档分块] → 块1: "事务ACID特性..."
           块2: "MVCC实现原理..."
           块3: "锁机制详解..."
    ✅ 问题："什么是事务？" → 块1 相似度 0.92 → 命中！
```

### 3.2 分块参数

```java
块大小: 500 字符    // ≈ 200-300 tokens，语义完整的最小单位
重叠:   100 字符    // 防止边界断句造成信息丢失
```

**分块策略选择**：

| 策略 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| **固定大小** | 简单、速度快 | 可能切断语义 | TinyBrain Phase 2 |
| 语义分块 | 语义完整 | 需要 NLP 模型 | 生产环境 |
| 递归分块 | 自适应内容 | 实现复杂 | LangChain 默认 |
| Markdown 分块 | 结构清晰 | 依赖格式 | 技术文档 |

### 3.3 重叠(Overlap)的作用

```
块1: "Spring 事务的核心是[AOP代理和事务管理器。事务传播行为定义了]"
块2: "[事务传播行为定义了]方法之间的事务边界..."
                                      ↑ 重叠区域确保"事务传播行为"的关键信息不会丢失
```

## 四、向量与相似度计算

### 4.1 什么是向量嵌入？

文本 → Embedding 模型 → 浮点数数组（如 1024 维）

```
"Spring 事务" → [0.123, -0.456, 0.789, ..., 0.321]  (1024 维)
"MySQL 索引" → [0.654, -0.123, 0.234, ..., -0.456]  (1024 维)
```

语义相近的文本，向量之间的夹角越小（余弦相似度越高）。

### 4.2 余弦相似度公式

```
cos(θ) = (A·B) / (||A|| × ||B||)

A·B = A₁B₁ + A₂B₂ + ... + AₙBₙ    (点积)
||A|| = √(A₁² + A₂² + ... + Aₙ²)   (向量长度)
```

**面试题：三种相似度对比**

| 方法 | 公式 | 特点 |
|------|------|------|
| 余弦相似度 | cos(θ) = (A·B)/(\ | A\ | \ | B\ | ) | 关注方向，忽略长度（推荐） |
| 欧氏距离 | d = √Σ(Aᵢ-Bᵢ)² | 关注绝对距离 |
| 点积 | A·B = ΣAᵢBᵢ | 受向量长度影响大 |

### 4.3 向量检索方式

| 方式 | 原理 | 速度 | 精度 | 适用场景 |
|------|------|------|------|---------|
| **全量扫描**（TinyBrain 当前） | 计算所有向量余弦值 | O(n) | 精确 | 数据量 < 10万 |
| IVF | 聚类后只在最近簇搜索 | O(log n) | 近似 | 大规模 |
| HNSW | 分层可导航小世界图 | O(log n) | 近似 | 高维大规模 |
| PQ | 乘积量化压缩向量 | O(1) | 近似 | 极致性能 |

## 五、LLM API 设计

### 5.1 为什么用 DeepSeek？

| 对比项 | DeepSeek | OpenAI GPT-4 |
|--------|---------|-------------|
| 价格 | ≈ ¥1/百万Token | ≈ ¥50/百万Token |
| 中文能力 | ✅ 优秀（中文训练语料丰富） | ⚠️ 一般 |
| API 兼容 | ✅ OpenAI 格式 | ✅ 标准 |
| Embedding | ✅ 支持 | ✅ 支持 |

### 5.2 API 调用设计

```java
// Chat 对话
POST https://api.deepseek.com/v1/chat/completions
{
  "model": "deepseek-chat",
  "messages": [
    {"role": "system", "content": "你是一个知识助手"},
    {"role": "user", "content": "Spring事务原理"}
  ],
  "temperature": 0.7
}

// Embedding 向量化
POST https://api.deepseek.com/v1/embeddings
{
  "model": "deepseek-embedding",
  "input": ["Spring事务原理是什么"]
}
```

**配置方式**：API Key 通过环境变量注入，不硬编码
```yaml
tinybrain:
  llm:
    api-key: ${TINYBRAIN_LLM_KEY}
    base-url: https://api.deepseek.com
```

## 六、RAG Prompt 工程

### 6.1 Prompt 结构

```
System: 你是一个知识库助手，请基于提供的上下文回答问题。
        如果上下文不足以回答，如实说明。请注明信息来源。

User:
请根据以下上下文回答问题。

=== 上下文 ===
【来源：Spring Boot 自动配置原理】
Spring Boot 通过 @EnableAutoConfiguration ...
【来源：AOP 实战】
AOP 在 Spring 中通过动态代理实现...

=== 问题 ===
Spring 事务是怎么实现的？

=== 要求 ===
请用中文回答，并在回答末尾列出信息来源。
```

### 6.2 优化技巧

| 技巧 | 说明 | 效果 |
|------|------|------|
| 来源标注 | 每个块前加【来源：文档标题】 | LLM 知道出处，减少幻觉 |
| 长度控制 | 上下文不超过模型窗口的 1/3 | 避免注意力分散 |
| 拒绝模板 | "信息不足时请如实说" | 减少幻觉 |
| 排序优化 | 高相关度块放前面 | 提升回答质量 |

## 七、面试题精讲

### 基础题

1. **RAG 是什么？解决了什么问题？**
   > RAG = Retrieval Augmented Generation。解决 LLM 知识截止和幻觉问题。

2. **RAG 的全流程是怎样的？**
   > 分块 → 向量化 → 存储 → 检索 → 增强 → 生成

3. **为什么需要文档分块？不分块行吗？**
   > 不分块：整篇文档一个向量，查询特异性不够，无法精确定位。

### 进阶题

4. **分块大小怎么确定？**
   > 取决于 Embedding 模型的最大 Token 限制和业务场景。常见 256-512 tokens。

5. **余弦相似度和欧氏距离的区别？在什么场景选哪个？**
   > 余弦看方向（语义）、欧氏看距离（数值）。文本检索通常用余弦。

6. **怎么评估 RAG 效果？**
   > 检索质量：Recall@K、MRR。生成质量：人工评估、GPT 评估。

### 场景题

7. **"知识库里有100万块，检索变慢了怎么办？"**
   > 内存向量库 → 替换为 Milvus/ChromaDB。全量扫描 → 替换为 HNSW 索引。

8. **"检索出来的上下文不相关怎么办？"**
   > 优化分块策略、换更好的 Embedding 模型、加重新排序(Rerank)环节。

9. **"用户问的问题不在知识库里怎么办？"**
   > 设置拒绝模板："根据提供的知识无法回答"。加一个退路：用 LLM 自身知识给出答案并声明风险。
