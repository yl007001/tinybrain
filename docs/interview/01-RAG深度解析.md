# RAG 系统面试深挖

## 一、项目中的实现（TinyBrain）

### 当前架构

```
用户问题 → 查询改写(LLM) → 向量化(Embedding API)
    → IVF ANN 检索(≈O(√n)) + 暴力搜索回退(O(n))
    → Top-K 结果融合 → Prompt 拼接 → LLM 生成回答
    → 流式 SSE 推送
```

### 关键技术选型
| 组件 | 选型 | 选型理由 |
|------|------|---------|
| 向量索引 | IVF (Inverted File) | K-means 聚类 + nprobe 探测，比暴力搜索快 10-50x |
| 向量存储 | JSON 文件持久化 | 轻量级，无需额外基础设施，重启不丢数据 |
| 嵌入模型 | DeepSeek Embedding (1024维) | OpenAI 兼容格式，批量嵌入支持 |
| 检索策略 | IVF + 暴力搜索融合 | 保证召回率（IVF 可能漏掉边界附近的结果） |
| 查询改写 | LLM (DeepSeek Chat) | 模糊问题 → 清晰表述，提升检索命中率 |

## 二、面试必问题

### Q1: 为什么用余弦相似度？不用欧氏距离？
- 余弦相似度关注方向而非模长，适合文本语义
- 欧氏距离受向量模长影响大，文本长度不同会导致模长方差大
- 余弦相似度值域 [-1,1]，更容易设置阈值（0.3）
- 面试引申：归一化后两者等价

### Q2: 如果有一百万条向量，搜索性能会怎样？
- 暴力搜索：O(n) = 100万次余弦计算 ≈ 50ms（1024维 × 100万）
- IVF 搜索：O(K + n/K × nprobe) ≈ K=1000, nprobe=10 → 1000 + 100万/1000×10 × 1024 ≈ 11K FLOPS
- 加速比：约 50-100x
- 代价：召回率从 100% 降到 ~90%（可通过融合暴力搜索弥补）

### Q3: 如何评估检索质量？
- MRR (Mean Reciprocal Rank)：正确答案出现在第几位？MRR = 1/rank
- Recall@K：前 K 个结果中包含多少正确答案
- NDCG：考虑排名的位置权重
- 示例：100 条测试数据，IVF Recall@5 = 0.92，BruteForce = 1.0

### Q4: RAG 的常见优化手段有哪些？
1. **查询改写** (已实现)
2. **HyDE** (Hypothetical Document Embedding)：先让 LLM 生成假设答案，用假设答案去检索
3. **多查询融合**：生成多个查询变体，分别检索后融合
4. **重排序** (Reranking)：先用向量检索 Top-50，再用 Cross-Encoder 精细排序
5. **分块策略优化**：语义分块、递归分块、Markdown 结构分块

### Q5: Embedding API 挂了怎么办？
1. **熔断保护**：Resilience4j CircuitBreaker 3次失败后断开 10s
2. **重试**：最多 2 次重试，间隔 1s
3. **降级**：返回友好提示 "AI 服务暂不可用"
4. **本地回退**：支持 Ollama 本地模型

## 三、进阶考点

### 索引构建策略
- IVF 的 K 取值：经验值 K ≈ √(N×10)，本项目中自动计算
- K-means 初始化：K-means++ 比随机初始化更稳定（避免空簇）
- 重建时机：首次向量数 > 1000 时触发，后续每 30s 检查

### 融合搜索策略
```
用户查询
  ├→ IVF ANN 搜索 (K=256, nprobe=4) → 候选集 A
  └→ 暴力搜索 (parallelStream) → 候选集 B
       ↓
  A ∪ B → 去重 → 按分数排序 → Top-K
```

### 批量嵌入优化
- 单次 API 调用发送多个文本，减少 HTTP 开销
- API 调用次数从 O(N) 降为 O(1)
- 对于 100 个 chunk，从 100 次 API 调用降为 1 次
