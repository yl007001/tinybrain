# 系统设计面试准备

## 一、从 TinyBrain 看微服务设计

### 当前架构：模块化单体 (Modular Monolith)

```
tinybrain-app (单体 JAR)
├── tinybrain-common (配置、异常、工具类)
├── tinybrain-user (认证、权限)
├── tinybrain-knowledge (文档 CRUD)
├── tinybrain-rag (向量检索、LLM 调用)
├── tinybrain-agent (Function Calling)
└── tinybrain-gateway (独立网关服务)
```

### 面试回答：如果让你拆成真正微服务

```
用户服务 (user-service)      → 独立 JAR，独立 MySQL 库
知识库服务 (knowledge-service) → 独立 JAR，独立 MySQL 库
RAG 服务 (rag-service)       → 独立 JAR，独立向量库
Agent 服务 (agent-service)   → 独立 JAR，独立会话存储
网关 (gateway)              → Spring Cloud Gateway + Nacos

通信方式：REST (同步) + RocketMQ (异步)
服务发现：Nacos
配置中心：Nacos
分布式事务：Seata AT / TCC
```

## 二、分布式系统必问题

### Q1: 分布式事务怎么做？
- 场景：创建文档 → 索引 RAG（文档落 MySQL + 向量落 VectorStore）
- 两阶段不能保证一致性
- 解决方案： 
  1. **本地消息表**：先写 MySQL，定时任务扫表异步索引
  2. **Seata AT**：自动回滚反向 SQL
  3. **事务消息**：RocketMQ 半消息

### Q2: 缓存策略？
- RAG 结果缓存：Redis TTL 5分钟
- 文档列表缓存：Redis TTL 5分钟
- Embedding 缓存：相同文本的向量缓存起来（MD5 作为 key）
- 缓存更新策略：Cache-Aside（查询时更新，写入时删除）

### Q3: 高并发场景怎么扛？
- **水平扩展**：无状态应用，加实例即可
- **性能瓶颈**：LLM API 调用（外部依赖，不可控）
- **优化手段**：
  - RAG 结果缓存（减少 LLM 调用）
  - 异步索引（文档上传后立即返回）
  - Embedding 缓存（相同问题直接命中）
  - 限流（Resilience4j RateLimiter 保护 LLM API）

## 三、技术深度必问题

### Q1: JWT vs Session？
- TinyBrain 用 JWT（无状态），适合微服务
- 优点：水平扩展无需 session 同步
- 缺点：无法主动吊销
- 改进：引入 Redis 黑名单 + 短期 Token+Refresh Token

### Q2: MyBatis-Plus 分页原理？
- PaginationInnerInterceptor 拦截 SQL
- 自动拼接 SELECT COUNT(*) 和 LIMIT/OFFSET
- 缓存 COUNT 结果（大表优化）

### Q3: Docker 多阶段构建的优势？
- 构建阶段：Maven + JDK 17 完整镜像
- 运行阶段：JRE 17 Alpine 精简镜像（~100MB vs ~300MB）
- 构建产物只保留 JAR，构建工具不留在运行镜像
