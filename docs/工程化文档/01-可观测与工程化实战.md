# Phase 5：可观测与工程化实战

## 一、为什么需要可观测性？

### 1.1 从"能跑"到"能诊断"

单体开发阶段，出问题可以通过本地日志 + IDE 调试解决。微服务化之后，一次请求可能跨越 3-5 个服务：

```
用户请求 → Gateway → User Service → RAG Service → Agent Service → LLM API
```

任何一个环节出问题，如果没有可观测性，排查就像大海捞针。

### 1.2 可观测性三大支柱

| 支柱 | 英文 | 作用 | TinyBrain 实现 |
|------|------|------|---------------|
| **日志** | Logging | 记录离散事件 | Logback + JSON + MDC |
| **指标** | Metrics | 聚合统计数据 | Micrometer + Prometheus |
| **链路追踪** | Tracing | 还原请求全链路 | Micrometer Tracing + Zipkin |

**Metrics vs Logging vs Tracing 的区别**
> Metrics 回答"系统有多慢"，Logging 回答"具体哪里出错了"，Tracing 回答"这个请求经过了哪些服务"。

## 二、结构化日志系统

### 2.1 为什么不用默认日志？

Spring Boot 默认日志是文本格式的：
```
2026-05-29 23:30:00.123 DEBUG [http-nio-8080-exec-1] c.t.rag.service.RAGService : 检索到 5 条相关结果
```

这种格式的问题：
- **机器不可读**：ELK 无法自动解析
- **缺少上下文**：不知道哪个用户、哪个请求
- **搜索困难**：grep 只能搜关键词，无法按字段过滤

### 2.2 JSON 日志格式

TinyBrain 使用 Logstash Logback Encoder 输出 JSON 格式：

```json
{
  "@timestamp": "2026-05-29T23:30:00.123+08:00",
  "level": "DEBUG",
  "logger": "com.tinybrain.rag.service.RAGService",
  "message": "检索到 5 条相关结果",
  "app": "tinybrain-app",
  "env": "prod",
  "mdc": {
    "traceId": "abc123def456",
    "userId": "42",
    "requestId": "xyz789"
  }
}
```

### 2.3 MDC 上下文注入

MDC (Mapped Diagnostic Context) 是 SLF4J 提供的线程级上下文，自动出现在日志中。

```java
// TraceIdFilter.java — 为每个请求注入 MDC
@Component
@Order(Ordered.HIGHEST_PRECEDENCE + 5)
public class TraceIdFilter implements Filter {
    public void doFilter(ServletRequest request, ServletResponse response,
                         FilterChain chain) {
        // 注入 traceId
        MDC.put("traceId", UUID.randomUUID().toString());
        MDC.put("requestId", UUID.randomUUID().toString());

        // 从请求头获取用户 ID（由 Gateway 注入）
        String userId = httpReq.getHeader("X-User-Id");
        if (userId != null) MDC.put("userId", userId);

        chain.doFilter(request, response);

        // 请求结束清理，防止内存泄漏
        MDC.clear();
    }
}
```

**MDC 和 ThreadLocal 的关系？**
> MDC 底层基于 ThreadLocal 实现，每个线程独立的日志上下文。正因为是 ThreadLocal，所以异步任务需要手动传递。

### 2.4 Logback 多环境配置


## 三、指标监控 (Micrometer + Prometheus)

### 3.1 Micrometer 是什么？

Micrometer 是 Java 生态的事实标准指标采集库。类似于 SLF4J 之于日志：
> SLF4J 是日志的门面，Micrometer 是指标的门面。

| 后端 | 协议 | 用途 |
|------|------|------|
| Prometheus | Pull (HTTP scrape) | 指标采集 + 告警 |
| Grafana | Push (HTTP POST) | 可视化大盘 |
| JMX | MBean | JVM 监控 |
| CloudWatch | AWS SDK | 云原生监控 |

### 3.2 @Timed 注解

```java
@Service
public class RAGService {
    @Timed(value = "rag.ask.time", description = "RAG 问答耗时",
           percentiles = {0.5, 0.95, 0.99})
    public RAGResult ask(String question, int topK) {
        // ...
    }
}
```

`@Timed` 自动生成以下 Prometheus 指标：
```
rag_ask_time_seconds_count{...}  # 调用次数
rag_ask_time_seconds_sum{...}    # 总耗时
rag_ask_time_seconds_max{...}    # 最大耗时
rag_ask_time_seconds_percentile_{0.5,0.95,0.99}{...}  # 百分位
```

**为什么要关注 P99 而不是平均值？**
> 平均值会被少数慢请求拉高，掩藏真实性能问题。P99（99% 的请求都在此时间内完成）更能反映用户体验。

### 3.3 自定义业务指标

除了 @Timed，还可以用 MeterRegistry 手动注册指标：

```java
@Service
public class MetricsService {
    private final Counter ragQueryCounter;
    private final Counter agentCallCounter;

    public MetricsService(MeterRegistry registry) {
        this.ragQueryCounter = Counter.builder("rag.queries.total")
                .description("RAG 查询总次数")
                .register(registry);
        this.agentCallCounter = Counter.builder("agent.calls.total")
                .description("Agent 调用总次数")
                .register(registry);
    }
}
```

### 3.4 Prometheus 采集配置

```yaml
# prometheus.yml
scrape_configs:
  - job_name: 'tinybrain-app'
    metrics_path: '/actuator/prometheus'
    static_configs:
      - targets: ['tinybrain-app:8080']
```

### 3.5 Grafana 大盘

Docker Compose 中预置了 Grafana 自动配置：
- Prometheus 数据源自动连接
- 可以通过导入社区 Dashboard（JVM Micrometer 等）快速搭建

## 四、分布式链路追踪 (Micrometer Tracing)

### 4.1 为什么需要链路追踪？

微服务环境下，一个请求的完整调用链可能包括多个服务：

```
Gateway → User(Dubbo) → Knowledge(MySQL) → Agent(RAG + LLM)
```

问题排查：
- ❌ 看每个服务的日志，手工串联
- ✅ Trace ID 自动贯穿所有服务，一键关联

### 4.2 Trace 和 Span

```
Trace: 用户注册流程
        ├── Span 1: Gateway 鉴权 (3ms)
        ├── Span 2: UserService 创建用户 (15ms)
        │   ├── Span 2.1: MySQL INSERT (5ms)
        │   └── Span 2.2: Redis 缓存 (2ms)
        ├── Span 3: Email 通知 (50ms)
        └── Span 4: 返回响应 (1ms)
```

| 概念 | 说明 | Logger MDC 中的字段 |
|------|------|---------------------|
| Trace ID | 贯穿整条请求链路的唯一 ID | `traceId=abc123` |
| Span ID | 当前操作（服务/方法）的唯一 ID | `spanId=def456` |
| Parent Span ID | 父级 Span，构成调用树 | - |

### 4.3 配置

```yaml
management:
  tracing:
    sampling:
      probability: 1.0  # 开发环境全量采样
    propagation:
      type: w3c         # W3C TraceContext 标准
```

跨服务传递时，Gateway 自动在 HTTP 头中携带 `traceparent`：

```
traceparent: 00-0af7651916cd43dd8448eb211c80319c-b7ad6b7169203331-01
```

### 4.4 Zipkin 可视化

启动后访问 http://localhost:9411：

```
[查询] 按 Service / Tag / Duration 过滤
[依赖图] 查看服务之间的调用拓扑
[详情] 查看每个 Span 的耗时和异常
```

## 五、容器化部署 (Docker Compose)

### 5.1 多阶段构建

```dockerfile
# 第一阶段：编译
FROM maven:3.9-eclipse-temurin-17 AS builder
WORKDIR /build
COPY . .
RUN mvn clean package -DskipTests

# 第二阶段：运行（仅含 JRE，体积缩小 70%+）
FROM eclipse-temurin:17-jre-alpine
COPY --from=builder /build/tinybrain-app/target/*.jar ./app.jar
ENTRYPOINT ["java", "-jar", "app.jar"]
```

### 5.2 Docker Compose 覆盖场景

TinyBrain 的 `docker-compose.yml` 包含 8 个服务：

| 服务 | 用途 | 端口 |
|------|------|------|
| mysql | 业务数据库 | 3306 |
| elasticsearch | 全文检索 | 9200 |
| redis | 缓存 + 限流 | 6379 |
| prometheus | 指标采集 | 9090 |
| grafana | 可视化大盘 | 3000 |
| zipkin | 链路追踪 | 9411 |
| tinybrain-app | 主应用 | 8080 |
| tinybrain-gateway | API 网关 | 8088 |

### 5.3 健康检查

```yaml
healthcheck:
  test: ["CMD", "curl", "-f", "http://localhost:8080/actuator/health"]
  interval: 30s
  timeout: 5s
  retries: 3
```

Docker Compose 的 `depends_on: condition: service_healthy` 确保按依赖顺序启动。

## 六、CI/CD (GitHub Actions)

### 6.1 Pipeline 流程

```
Push → 检出代码 → JDK 17 设置 → Maven 编译
                                            ↓
                                   运行单元测试
                                            ↓
                                  上传测试报告
                                            ↓
                                   Maven 打包
                                            ↓
                                  Docker 镜像构建 (main)
```

### 6.2 缓存加速

```yaml
- name: Setup JDK 17
  uses: actions/setup-java@v4
  with:
    java-version: 17
    distribution: 'temurin'
    cache: maven  # 缓存 ~/.m2/repository
```

### 6.3 质量门禁

```yaml
on:
  pull_request:
    branches: [main]
```

每次 PR 自动运行编译 + 测试，确保：
- ✅ 编译通过
- ✅ 单元测试全部通过
- ✅ 不引入新缺陷

## 七、测试策略

### 7.1 分层测试

| 类型 | 重点 | 占比 | TinyBrain 示例 |
|------|------|------|---------------|
| 单元测试 | Service/Util 逻辑 | 70% | JwtUtilTest, VectorStoreTest |
| 集成测试 | 组件协作 | 20% | TinyBrainApplicationTest |
| 端到端测试 | 完整流程 | 10% | (CI 环境执行) |

### 7.2 关注点

```java
// 测试 JWT Token 的生成和解析一致性
@Test
void generateAndParse_shouldBeConsistent() {
    String token = JwtUtil.generateToken(99L, "ROLE_USER");
    assertEquals(99L, JwtUtil.getUserId(token));
    assertTrue(JwtUtil.validateToken(token));
}

// 测试向量检索的正确性
@Test
void upsertAndSearch_shouldReturnTopK() {
    vectorStore.upsert(1L, 10L, List.of(1.0, 0.0, 0.0));
    List<SearchResult> results = vectorStore.search(query, 5);
    assertEquals(1, results.size());
    assertEquals(1L, results.get(0).getChunkId());
}
```
